1. INTRODUCTION
================================================================================

What is Smart Finance Tracker?
-------------------------------

Smart Finance Tracker is a complete personal finance management web 
application that I built using Python and Streamlit. It is a powerful yet 
easy-to-use tool that helps people track their money, manage their budgets, 
set financial goals, and even practice investment trading in a simulated 
environment.

The application allows users to record their income and expenses, categorize 
their spending, track their wallet balance and bank accounts, set monthly 
budgets for different spending categories, create savings goals with target 
amounts and deadlines, and practice buying and selling various investment 
assets like stocks, mutual funds, and cryptocurrencies in a risk-free 
simulated market.

Why This Project Was Built?
---------------------------

I decided to build this project because I noticed that many people struggle 
with managing their personal finances. Most of us receive our salaries or 
income but find it difficult to track where our money goes. We often wonder 
at the end of the month why we have so little savings despite earning a decent 
income.

Traditional spreadsheet tracking can be tedious and error-prone. There are 
many expensive finance apps available, but they often have complex features 
that confuse average users. I wanted to create something simple, intuitive, 
and accessible that anyone could use without any technical knowledge.

Real-Life Problem It Solves
----------------------------

This project addresses several real-life financial management problems:

First, it solves the problem of not knowing where money goes. When users 
record every expense, they can see exactly how much they spend on food, 
transportation, entertainment, and other categories. This awareness is the 
first step towards better financial management.

Second, it helps with budget overspending. Users can set monthly limits for 
different spending categories. The application alerts them when they are 
approaching or exceeding their budget, helping them make better spending 
decisions in real-time.

Third, it addresses the challenge of saving for goals. Whether it's buying 
a car, planning a vacation, or building an emergency fund, users can create 
specific financial goals with target amounts and track their progress over 
time.

Fourth, it provides a safe environment to learn about investments. Many 
people are interested in stock market investing but are afraid of losing 
money. The simulated market allows users to practice trading without any 
real risk, helping them understand how markets work.

Finally, it gives users a complete picture of their finances. By combining 
wallet balance, bank accounts, investments, and financial goals, users can 
see their overall net worth and make informed decisions about their money.

2. TECHNOLOGIES USED
================================================================================

Python
------

I chose Python as the primary programming language for this project for many 
reasons. Python is one of the most popular and beginner-friendly programming 
languages in the world. It has a clean, readable syntax that makes code easy 
to understand and maintain.

Python has an enormous ecosystem of libraries and frameworks that make 
development faster and easier. For this project, I needed libraries for 
database handling, web interface creation, password security, and data 
visualization, and Python provided all of these.

Python is also widely used in the fintech industry, so learning it has 
great career value. Many real-world finance applications, from banking 
systems to trading platforms, are built with Python.

Streamlit
---------

Streamlit is a Python library that allows developers to create beautiful, 
interactive web applications without needing to know HTML, CSS, or 
JavaScript. It transforms Python scripts into responsive web apps in 
minutes.

I chose Streamlit because it is perfect for data-driven applications like 
a finance tracker. It provides built-in support for forms, tables, charts, 
and user inputs, which are all essential for this project.

With Streamlit, I could focus on the business logic rather than worrying 
about web development complexities. The hot-reload feature during 
development means changes appear instantly, making debugging and 
iteration very fast.

Streamlit also handles session state management, which is crucial for 
maintaining user login sessions and storing temporary data as users 
navigate through different pages of the application.

SQLite
------

SQLite is a lightweight, file-based database that requires no separate 
server to run. It stores all data in a single file called fintech.db, 
making it easy to backup, transfer, and manage.

For a personal finance application like this, SQLite is perfect because 
it handles all the database operations efficiently without requiring 
complex setup. It supports standard SQL queries, transactions, and 
integrity constraints.

SQLite is reliable and has been extensively tested over decades. It uses 
file locking to ensure that data remains consistent even if multiple 
users access it simultaneously. For application, 
it provides all the features a single-user personal needed without the overhead of larger 
database systems.

I designed a comprehensive database schema with multiple tables for users, 
transactions, budgets, goals, investments, and various other data needed 
by the application.

bcrypt
------

bcrypt is a professional-grade password hashing library that implements 
the bcrypt algorithm. Password hashing is essential for security because 
it ensures that even if someone gains access to the database, they cannot 
read the actual passwords.

When a user creates an account or changes their password, bcrypt adds 
a random "salt" to the password and hashes it multiple times using the 
bcrypt algorithm. This produces a secure hash that cannot be reversed 
to get the original password.

The library also provides functions to verify passwords by hashing the 
entered password and comparing it with the stored hash. This is how login 
authentication works securely.

I chose bcrypt because it is widely recognized as one of the most secure 
password hashing methods available. It is used by major websites and 
applications worldwide.

Plotly
------

Plotly is a powerful data visualization library that creates interactive 
charts and graphs. I used Plotly to create various charts in the analytics 
section of the application.

Plotly charts are interactive, meaning users can hover over data points 
to see exact values, zoom in on specific areas, and toggle different 
datasets on and off. This makes financial data much easier to understand 
compared to static tables.

For this project, I used Plotly to create pie charts showing expense 
breakdown by category, bar charts comparing income versus expenses, 
line charts showing spending trends over time, and other visualizations 
that help users understand their financial patterns.

Plotly integrates seamlessly with Streamlit, making it easy to embed 
these interactive charts directly into the web application.

3. PROJECT FLOW
================================================================================

Let me explain how the application works from the moment it starts to when 
users complete various financial tasks:

App Start
---------

When users run the application, the main app.py file is executed. This file 
sets up the Streamlit page configuration, including the title "Smart Finance 
Tracker," the money emoji icon, and the wide layout for better visibility.

The application first calls the init_session_state() function, which 
initializes important session variables. These variables track whether the 
user is authenticated, their user information, admin information, and the 
current page they are viewing.

After initialization, the application checks if the user is authenticated. 
If not, it displays the login page. This is the first screen that users 
see when they open the application.

Login Page
----------

The login page offers three options: User Login, Registration, and Admin 
Login.

For regular users, they can either log in with their existing credentials 
or register a new account. The registration form collects username, email, 
mobile number, and password. The system validates all inputs to ensure 
data quality and security.

The admin login is for administrators who need to manage the platform, 
view user statistics, and monitor system activities.

Once users successfully log in, the application stores their information 
in session state and redirects them to the main dashboard.

Dashboard
---------

After logging in, users arrive at the Dashboard, which is their home base 
in the application. The dashboard provides a quick overview of their 
financial health.

It shows their current wallet balance, investment portfolio value 
(with profit/loss indicator), and total net worth by combining wallet 
and investment amounts.

The dashboard also displays a monthly summary showing income, expenses, 
and net savings for the current month. A savings rate indicator shows 
whether the user has good savings (20%+), low savings, or is 
overspending, giving instant feedback on financial habits.

Recent transactions are displayed at the bottom, showing the latest 
expenses with date, category, amount, and merchant details.

Wallet
------

The Wallet section allows users to manage their primary wallet and add 
or view bank accounts. They can add income to their wallet from various 
sources like salary, freelance work, or gifts.

Users can also add expenses directly from their wallet. When they do so, 
the application automatically deducts the amount from their balance and 
ensures they cannot spend more than they have.

Bank account management allows users to link multiple bank accounts 
and track balances across all of them. This helps users see their 
complete liquid assets in one place.

Transactions
------------

The Transactions page provides a detailed view of all financial activities. 
Users can see a chronological list of all income and expenses with 
filters for date range, category, and transaction type.

This page helps users review their spending history and identify patterns. 
They can search for specific transactions and export data if needed.

Budgets
-------

In the Budgets section, users can set monthly spending limits for various 
categories like food, transportation, entertainment, shopping, and others.

When users add expenses in any category, the application automatically 
tracks spending against the budget. If users approach or exceed their 
budget, they receive notifications and see visual warnings.

This feature helps users develop disciplined spending habits and avoid 
overspending.

Goals
-----

The Goals section allows users to create financial goals with target 
amounts and target dates. Examples include saving for a vacation, buying 
a car, or building an emergency fund.

Users can allocate money towards their goals from their wallet. The 
application tracks progress and shows how much more they need to save 
to reach each goal.

A key feature is the dynamic monthly contribution calculation. When a 
goal is created, the system calculates the required monthly savings 
based on the target amount and timeline. As users add more or less 
money than planned each month, the monthly contribution automatically 
recalculates based on the remaining amount and remaining months until 
the target date. This ensures users always see an accurate picture of 
how much they need to save going forward.

This feature helps users stay motivated by showing visual progress bars 
and gauge charts, and celebrating with balloons when they achieve their 
goals.

Investments
-----------

The Investments section provides a simulated market where users can 
practice investing without real risk. The market includes various assets 
like stocks, mutual funds, ETFs, and cryptocurrencies.

Users can view current market prices, which change dynamically based on 
simulated market movements. They can buy assets using their wallet 
balance and sell them to realize profits or losses.

The portfolio view shows all current holdings, total invested amount, 
current value, and profit or loss. This helps users understand how 
diversification works and learn about different asset types.

Analytics
---------

The Analytics page provides deep insights into spending patterns and 
financial behavior. Users can see charts and graphs showing expenses 
by category, income versus expense trends over time, daily spending 
patterns, and top expenses.

This data-driven analysis helps users make informed decisions about 
their finances and identify areas where they can improve.

Settings
--------

The Settings section allows users to manage their profile and change 
passwords. They can view their account information, update their mobile 
number, and review login activity history. The security tab provides 
password change functionality with current password verification.

5. CLASS & METHOD FLOW
================================================================================

Understanding how classes and methods work together is important for 
understanding how the application functions. Let me explain the complete 
flow:

AuthService Class - Authentication Management
-----------------------------------------------

The AuthService class in auth_service.py handles everything related to 
user authentication. It is created as a singleton instance called 
auth_service that is used throughout the application.

The hash_password() method takes a plain text password and converts it 
into a secure hash using bcrypt. This hash is what gets stored in the 
database instead of the actual password.

The verify_password() method takes a plain text password and compares 
its hash with the stored hash. If they match, the password is correct.

The register_user() method handles new user registration. It validates 
all inputs (username, email, mobile, password), checks if the user 
already exists, hashes the password, and creates the user record in 
the database.

The login_user() method authenticates users. It finds the user by email, 
verifies the password, checks if the account is locked, and returns 
user data if successful.

The login_admin() method works similarly but for administrator accounts.

How AuthService Connects to Database:
--------------------------------------

The AuthService uses the db object from database/db.py to interact 
with the database. For example, when registering a user, it calls:

    db.user_exists(username=username) - to check if username is taken
    db.create_user() - to create the new user record
    db.log_action() - to record the registration activity

The authentication flow works like this:

1. User enters email and password on the login page
2. app.py calls auth_service.login_user(email, password)
3. login_user() calls db.get_user_by_email() to find the user
4. login_user() calls verify_password() to check the password
5. If successful, login_user() updates last login time in database
6. The user data is returned to app.py
7. app.py stores user data in session_state
8. User is redirected to the dashboard

WalletService Class - Financial Operations
-------------------------------------------

The WalletService class handles all monetary transactions. It ensures 
that money is handled safely through database transactions.

The add_income() method adds money to the user's wallet or bank account. 
It validates the amount, converts it to paise, and uses a database 
transaction to update the balance and record the income atomically.

The add_expense() method records spending. It first checks if the user 
has sufficient balance, then deducts the amount and creates an expense 
record. It also checks budget limits and returns warnings if the user 
is overspending.

The get_balance() method calculates the total available balance across 
wallet and all bank accounts.

The get_budget_status() method checks how much of the monthly budget 
has been spent and returns percentage used.

InvestmentService Class - Investment Simulation
-------------------------------------------------

The InvestmentService class manages the simulated investment market.

The get_market_assets() method returns all available assets that users 
can invest in, including stocks, mutual funds, cryptocurrencies, and ETFs.

The buy_asset() method handles purchasing an investment. It verifies 
wallet balance, calculates the total cost, updates the user's portfolio, 
and records the transaction.

The sell_asset() method handles selling investments. It verifies the 
user owns sufficient units, calculates the proceeds, updates the portfolio, 
and records the transaction with profit or loss.

The update_market_prices() method simulates realistic price movements 
for all assets based on their volatility.

AnalyticsService Class - Data Analysis
--------------------------------------

The AnalyticsService class provides analytical insights.

The get_user_dashboard_data() method aggregates all data needed for 
the dashboard including balances, monthly summaries, recent transactions, 
and financial health score.

The calculate_financial_health() method analyzes multiple factors like 
savings rate, budget compliance, emergency fund status, and investment 
diversification to produce a health score out of 100.

6.2 USER LOGIN FLOW (Complete Call Chain)
================================================================================

When user enters email/password and clicks Login, here's exactly what 
happens:

USER FILLS FORM:
    User enters: email = "john@example.com", password = "SecurePass123"
    Clicks "Login" button
    
    |
    V
app.py: show_login_page() [Lines 120-136]
    |
    +-> Streamlit captures form input
    +-> Calls: success, message, user_data = auth_service.login_user(email, password)
    |
    V
services/auth_service.py: login_user() [Lines 151-175]
    |
    +-> STEP 1: Validate inputs not empty
    |
    +-> STEP 2: Look up user in database
    |   +-> Calls: user = db.get_user_by_email(email.lower())
    |       |
    |       +-> database/db.py: get_user_by_email() [Line 213]
    |           |
    |           +-> SQL Query: "SELECT * FROM users WHERE LOWER(email) = ?"
    |           +-> Parameter: (email.lower(),)
    |           +-> Returns: user dict or None
    |
    +-> STEP 3: If user NOT found in database:
    |   +-> Calls: self._log_login_attempt(email, False)  # Record failed attempt
    |   +-> Returns: (False, "Invalid email or password", None)
    |
    +-> STEP 4: If user IS found in database:
        +-> Check if account is locked:
        |   +-> if user['locked_until'] > current_time:
        |       return (False, "Account locked. Try again later", None)
        |
        +-> Verify password:
        |   +-> Calls: self.verify_password(password, user['password_hash'])
        |       |
        |       +-> Uses bcrypt.checkpw() function
        |       +-> bcrypt.hashpw(password.encode(), stored_hash)
        |       +-> Returns: True if match, False otherwise
        |
        +-> If password WRONG:
        |   +-> Increments failed_login_attempts counter in database
        |   +-> If attempts >= 5:
        |       +-> Sets locked_until = now() + 3 minutes
        |   +-> Returns: (False, "Invalid email or password", None)
        |
        +-> If password CORRECT:
            +-> Calls: db.update_last_login(user_id)  # Update last login time
            +-> Returns: (True, "Login successful!", user_data_dict)
    |
    V
BACK TO app.py: show_login_page()
    |
    +-> If login SUCCESS (success = True):
    |   +-> st.session_state.authenticated = True
    |   +-> st.session_state.user = user_data    # Store user dict in session
    |   +-> st.success(message)                  # Show success message
    |   +-> st.rerun()                           # Force Streamlit to re-run
    |
    +-> If login FAILS:
        +-> st.error(message)                   # Show error message

WHY THIS FLOW:
- We use bcrypt because it's secure (adds random salt, slow hash)
- We check account lockout to prevent brute force attacks
- We log every attempt for security auditing (audit_logs table)
- We use session state because Streamlit re-runs on each action

6.4 ADDING INCOME FLOW
================================================================================

When user adds money to wallet:

USER ACTION (pages/wallet.py):
    Fills: amount = 5000, source = "Salary", category = "Salary"
    Selects: "Wallet" as account
    Clicks "Add Income"
    
    |
    V
pages/wallet.py: show_wallet() [Lines 78-90]
    |
    +-> Gets user_id: user_id = st.session_state.user['user_id']
    +-> Calls: wallet_service.add_income(
    |       user_id=1, 
    |       amount=5000.0, 
    |       source="Salary", 
    |       category="Salary",
    |       account_type="WALLET",
    |       account_id=None,
    |       description=None
    |   )
    |
    V
services/wallet_service.py: add_income() [Lines 18-66]

    STEP 1: Validate input
    |
    +-> if amount <= 0: return False, "Amount must be positive", None
    +-> amount_paise = db.to_paise(amount)  # Convert rupees to paise
    |   +-> Formula: int(round(5000.0 * 100)) = 500000 paise
    |
    STEP 2: Start database transaction (atomic operation)
    |
    +-> with db.transaction() as conn:
    |   +-> Executes: BEGIN IMMEDIATE (locks database)
    |   +-> If error occurs: automatically ROLLBACK
    |   +-> If success: automatically COMMIT
    |
    STEP 3: Update wallet balance
    |
    +-> cursor.execute("SELECT wallet_balance FROM users WHERE user_id = ?", (1,))
    +-> current_balance = 10000  (example)
    +-> new_balance = 10000 + 500000 = 510000 paise
    |
    +-> cursor.execute("UPDATE users SET wallet_balance = ? WHERE user_id = ?", 
    |   (510000, 1))
    |
    STEP 4: Record wallet transaction
    |
    +-> cursor.execute("""INSERT INTO wallet_transactions 
    |   (user_id, txn_type, amount, balance_after, reference_type, description)
    |   VALUES (?, 'INCOME', ?, ?, 'income', ?)""",
    |   (1, 500000, 510000, "Salary"))
    |
    STEP 5: Log action and return
    |
    +-> db.log_action('USER', 1, 'Income added', 'wallet', 1, None, {'amount': 5000})
    +-> return (True, "Income added successfully!", {'new_balance': 510000})

WHY TRANSACTIONS:
- Prevents data inconsistency if something fails mid-operation
- Either ALL changes happen (balance + transaction record) or NONE happen
- If balance updates but transaction record fails, we rollback - no orphan data

6.11 KEY DESIGN PATTERNS USED
================================================================================

1. SINGLETON PATTERN - Database
   The Database class uses singleton pattern so only one connection pool exists:
   
       class Database:
           _instance = None  # Class variable - shared by all
           
           def __new__(cls, ...):
               if cls._instance is None:
                   cls._instance = super().__new__(cls)
               return cls._instance
   
   This ensures all parts of the app use the same database connection.

2. SERVICE LAYER PATTERN
   We separate business logic from UI:
   - pages/*.py       = UI (what user sees and interacts with)
   - services/*.py   = Business logic (how it works)
   - database/db.py  = Data access (how we store/retrieve)
   
   This makes code reusable across pages and easier to test.

3. TRANSACTION PATTERN
   For financial operations, we use atomic transactions:
   
       with db.transaction() as conn:
           conn.execute("UPDATE users SET balance = ? ...", (new_balance,))
           conn.execute("INSERT INTO transactions ...", (...))
       # If error: automatically rolled back
       # If success: automatically committed
   
   This prevents partial updates (e.g., balance deducted but no record).

4. SESSION STATE PATTERN
   Streamlit reruns the script on every interaction. We use session state to:
   - Keep user logged in between reruns
   - Store current page name
   - Cache expensive calculations
   
       if 'user' not in st.session_state:
           st.session_state.user = None

6.12 SECURITY MEASURES IMPLEMENTED
================================================================================

1. PASSWORD SECURITY
   - Passwords are NEVER stored in plain text
   - bcrypt with 12 rounds of salting
   - Slow hash prevents brute force attacks
   
   Code: auth_service.py
       salt = bcrypt.gensalt(rounds=12)
       hash = bcrypt.hashpw(password.encode(), salt)

2. SQL INJECTION PREVENTION
   - NEVER use string formatting for SQL
   - ALWAYS use parameterized queries with ? placeholders
   
   WRONG (vulnerable to SQL injection):
       query = f"SELECT * FROM users WHERE email = '{email}'"
   
   RIGHT (safe):
       query = "SELECT * FROM users WHERE email = ?"
       cursor.execute(query, (email,))

3. USER DATA ISOLATION
   - Every query includes user_id filter
   - Users can ONLY access their own data
   - Admin queries verify admin role

4. ACCOUNT LOCKING
   - After 5 failed login attempts, account locks for 3 minutes
   - Prevents brute force password attacks

5. AUDIT LOGGING
   - Every important action is logged to audit_logs table
   - Tracks: who (actor_id), what (action), when (created_at), old/new values
   - Useful for security investigation
